ERC-998 is the Composable Non-Fungible Token Standard on [[ethereum]], an extension of [[erc-721]] that allows a single NFT to own other NFTs ([[erc-721]] tokens) and fungible [[ethereum|ERC-20]] tokens, forming hierarchical ownership trees. Where [[erc-721]] treats each NFT as a standalone asset, ERC-998 treats an NFT as a container whose value can include the aggregate value of everything it holds. The standard has seen limited production adoption relative to ERC-721 and ERC-1155, but is used by wrapper-style contracts such as the [[mooncats|MoonCatAcclimator]] (Source: [[2026-04-22-gap-finder-the-provided-wiki-pages]]).

## Concept

ERC-998 defines two complementary composability models, which can be combined in a single token:

- **Top-down composables** — a parent ERC-998 NFT explicitly owns a set of child NFTs or fungible tokens. Ownership of the children is recorded in the parent's contract; transferring the parent transfers every child with it. The parent is the "cold" root of the ownership tree and queries about children flow down from it.
- **Bottom-up composables** — a child NFT or token records internally which parent NFT it is attached to. The child "points up" to its parent rather than the parent listing its children. This is useful when an existing NFT needs to be bound to a container after the fact without the parent contract tracking every possible child.

In practice, top-down models are the more common implementation because they keep enumeration and transfer logic localized in one contract. Bottom-up models reduce what the parent has to know but push more logic into each child. ERC-998 specifies both so that a composable NFT ecosystem can support arbitrary nesting in either direction.

## Technical model

ERC-998 is not a single monolithic interface — it is a family of ERC-721 sub-interfaces that can be mixed and matched:

- `ERC998ERC721TopDown` — a parent NFT that can own child ERC-721 tokens
- `ERC998ERC20TopDown` — a parent NFT that can own ERC-20 token balances
- `ERC998ERC721BottomUp` — an ERC-721 token that can be attached to a parent NFT
- `ERC998ERC20BottomUp` — an ERC-20 extension where balances can be attached to a parent NFT

Each sub-interface extends standard ERC-721 so that composable NFTs remain compatible with existing wallets and marketplaces that only understand [[erc-721]]. Ownership hierarchies are then represented as:

```
Composable Parent NFT (ERC-998)
├── Child NFT (ERC-721)
├── Child NFT (ERC-721)
│   └── Grandchild NFT (ERC-721, if itself composable)
└── ERC-20 balance: 1,000 TOKEN
```

Transferring the parent atomically transfers the entire subtree — a single on-chain transaction can move a bundle of assets that would otherwise require dozens of separate ERC-721/ERC-20 transfers. This is the core operational advantage: **bundled ownership, bundled transfer, single unit of trade**.

## Notable implementations

- **[[mooncats|MoonCatAcclimator]]** — the wrapping contract for [[mooncats]] deployed at `0xc3f733ca98E0daD0386979Eb96fb1722A1A05E69`. It makes the original MoonCatRescue tokens ERC-721 compatible for modern wallets and marketplaces while implementing ERC-998 composability, meaning a wrapped MoonCat can itself own other NFTs and ERC-20 balances (Source: [[2026-04-22-gap-finder-the-provided-wiki-pages]]). This was deployed by [[ponderware]] as part of the Acclimator wrapper system.
- **Crypto-Composables** — the reference-implementation project that accompanied the original ERC-998 draft, providing example contracts for both top-down and bottom-up models and serving as the canonical codebase that later implementers have adapted.

Beyond these, individual game and metaverse projects have experimented with ERC-998 for in-game item bundling (characters that own equipment NFTs, for example), though most mainstream NFT collections still rely on plain [[erc-721]] or ERC-1155.

## Trading relevance

Composability materially changes how NFT valuation and [[nft-trading]] work:

- **Derivative value from holdings** — a composable NFT is worth at least the sum of the assets it owns. Valuing the parent requires summing the floor prices of every held NFT plus the mark-to-market value of every held ERC-20 balance, then adding any premium for the parent itself. Traders evaluating a composable NFT cannot rely on collection floor alone.
- **Bundling as a strategy** — sellers can deliberately load value into a composable parent (e.g. stuff a wrapped MoonCat with complementary NFTs or tokens) to create a themed bundle that trades as a single unit. This can capture bundle premia if buyers value the thematic coherence, or avoid gas costs of selling pieces individually.
- **Unbundling as a strategy** — buyers who acquire an undervalued composable parent can extract children and sell them separately when the sum-of-parts exceeds the bundle price. This is structurally similar to closed-end fund arbitrage: if a composable NFT trades at a discount to the NAV of its contents, unwrapping captures the spread.
- **Gas-efficient portfolio transfer** — moving a composable NFT moves its entire subtree in one transaction, which is significant for large portfolios during periods of high gas.
- **Hidden-state risk** — because ownership is nested, a buyer who only inspects the parent's token URI may miss material changes to children between listing and execution. Front-running attacks where a seller drains a composable NFT's children before the sale completes are a known class of risk that ERC-721-only marketplaces do not natively guard against.

## Adoption status

As of 2026, ERC-998 remains a niche standard. Mainstream NFT infrastructure — including most wallets, indexers, and marketplaces — optimizes for [[erc-721]] and ERC-1155, and composable ownership trees are typically either ignored (treated as a plain parent NFT) or supported only by specialized tooling. The standard's primary production footprint is through wrapper contracts like the [[mooncats|MoonCatAcclimator]] and a small number of legacy projects rather than new collection launches (Source: [[2026-04-22-gap-finder-the-provided-wiki-pages]]).

Adoption has been limited for several practical reasons: composability adds contract complexity and audit surface; marketplace UX for inspecting nested holdings is underdeveloped; and most collection designs do not need nested ownership to express their mechanics. Where composability is useful, project teams have increasingly preferred off-chain "vault" patterns or ERC-1155 multi-token contracts over full ERC-998 conformance.
